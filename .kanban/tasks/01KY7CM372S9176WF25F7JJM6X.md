## What

Replace the mutable connect-slot event wiring (shipped in `feat(operations): add standard progress/completion event vocabulary…`) with **two small orthogonal, pure capability protocols**. The shipped design holds one mutable sink per tool instance (`OperationEventSinkHolder`), so two sessions sharing a tool instance (e.g. a Router session and its fork) steal event delivery from each other — last connect wins. The replacement: nothing is mutated after construction; hosts derive per-session tool instances by pure functions.

Land in this package:

**1. `EventEmittingTool` reworked to pure instancing:**
- Replace `func connect(_ sink:)` (mutation) with `func connecting(_ sink: any OperationEventSink) -> any Tool` (pure copy; the copy shares the receiver's underlying reference-typed state, only the event route differs). Docs keep the "hosts instantiate, users don't" contract: a host maps its `[any Tool]` list through `(tool as? any EventEmittingTool)?.connecting(sink) ?? tool` at session setup; conformance stays discoverable from an `any Tool` existential (no associated types).
- `EventEmittingContext`: replace the holder requirement with a plain immutable value — `var operationEventSink: (any OperationEventSink)? { get }` plus `func connecting(_ sink:) -> Self` (copy with sink replaced). Document the capture-at-start rule: operations that start long-running work capture the context's sink once, at operation start, into whatever outlives the call — that is what keeps event ownership with the session whose turn started the operation.
- **Delete `OperationEventSinkHolder`** (lock, `@unchecked Sendable`, replace semantics — all gone).

**2. New `ForkableTool` with a blanket value-semantics default:**
- `public protocol ForkableTool: Tool { func forked() -> any Tool }` — a child session's instance of this tool, derived at fork time. Blanket default in a protocol extension: `func forked() -> any Tool { self }` — for a struct tool this is a genuine new instance (copy); document the caveat that the default assumes value semantics and a class-based tool must override.
- `public protocol ForkableContext: Sendable { func forked() -> Self }` with the same blanket copy default. `OperationTool` (a struct) conforms to `ForkableTool` unconditionally: `forked()` returns a copy whose context is `(context as? any ForkableContext)?.forked() ?? context` — so fork semantics are context-owned and tool-internal; hosts never see the difference.
- Host composition contract (document, host implements): at fork, `((tool as? any ForkableTool)?.forked() ?? tool)` then bind events via `connecting` if emitting. Plain tools conform to neither and pass through shared.

**Shared plumbing:** `OperationTool.init` throws (schema fusion) — provide one non-throwing internal copy path (reusing the already-fused schema) that both `connecting` and `forked` use.

`OperationEvent` / `OperationEventSink` are unchanged. Rework `EventEmittingToolTests` (and add `ForkableToolTests`): route independence, state sharing across copies, blanket-default fork behavior, mixed-list host mapping; delete holder-specific tests.
